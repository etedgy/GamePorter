GamePorter (created 2026-07-06): personal-use native SwiftUI app replacing CrossOver/Whisky. Repo `~/GamePorter` → private GitHub `etedgy/GamePorter`, `./build.sh` → /Applications/GamePorter.app (ad-hoc signed).

- Runtime: `~/Library/Application Support/GamePorter/` — `Toolkit/` holds the Gcenx game-porting-toolkit 3.0-3 build (x86_64 Wine + Apple D3DMetal.framework, runs under Rosetta; wine at `Toolkit/Game Porting Toolkit.app/Contents/Resources/wine/bin/wine64`), `Bottles/<uuid>/` = Wine prefixes + `gameporter.json`, `Logs/`.
- App auto-downloads the toolkit from Gcenx GitHub releases if missing; SetupView also imports Apple's official GPTK dmg (redist/lib overlay) to upgrade D3DMetal — user has an Apple dev account.
- Features: bottles (win10 default), run exe (`wine64 start /unix`), .msi via msiexec, one-click Steam installer, program auto-discovery + pinning, toggles ESYNC / MTL_HUD_ENABLED / RetinaMode registry / ROSETTA_ADVERTISE_AVX, tools (winecfg/taskmgr/regedit/uninstaller), wineserver -k kill, uninstall (PR #1: parses system.reg Uninstall keys; QuietUninstallString → `msiexec /x{GUID} /qn` → cmd /c UninstallString → uninstaller --remove).
- Gotcha learned live: MSI UI dialogs (even /qb) fail to render under this Wine build and hang invisibly — always use /qn for msiexec; wine GUI processes inherit com.yan.gameporter bundle id.
- CrossOver-style resource handling (PR #2): WINE_LARGE_ADDRESS_AWARE=1 default env + RLIMIT_NOFILE raised to 65536 at app launch (children inherit; Finder apps default to 256). Installer-crash detection: scans wine log for `alloc_pages_vprot`/"Assertion failed" → surfaces plain-language message instead of silent quit.
- FitGirl-repack finding CORRECTED (2026-07-06): `alloc_pages_vprot` crash is a Wine 7.x bug FIXED in newer Wine — NOT a hard limit. CrossOver 26.1 uses Wine 11.0 and installs the repack fine (user confirmed). Verified: THPS setup.exe crashes on our GPTK Wine 7.7 but gets PAST the assertion on Gcenx wine-staging 11.10. Root cause was the stale engine, not memory env. User owns THPS 1+2 on Xbox (legit), couldn't get Mac version.
- Multi-engine + multi-renderer (PR #3): app now installs/holds multiple Wine engines side by side and picks engine+renderer PER BOTTLE. Engines under `Engines/<id>/` (+ legacy GPTK under `Toolkit/`). Catalog: Gcenx macOS_Wine_builds wine-staging-11.10 (newest, vanilla, DXMT/DXVK/WineD3D) + game-porting-toolkit 3.0-3 (GPTK, D3DMetal/DXVK/WineD3D). NOTE: Wine 11 loader binary is `wine` (not `wine64`); EngineManager.findLoader handles both. Renderers via WINEDLLOVERRIDES + RendererStager staging dlls into prefix (x86_64-windows→system32, i386-windows→syswow64): DXVK=Gcenx DXVK-macOS v1.10.3 (only d3d10core+d3d11, dxgi stays builtin), DXMT=3Shain/dxmt v0.80 (needs Wine 10.18+, winemetal.so via WINEDLLPATH to x86_64-unix dir, +MetalFX). Vanilla wine-staging does NOT bundle Apple D3DMetal (that's GPTK-only). Files: Models/Engine.swift, Managers/EngineManager.swift, Managers/RendererStager.swift, Views/EnginesView.swift; Bottle.engineID/renderer are optional (Codable-safe migration).
- Installer crashes on Wine 11 — 3 causes FIXED (PR #4): (a) game renderer (DXMT/DXVK) DLL overrides + winemetal forced onto 32-bit installers → `mmap() Cannot allocate memory` at high 0x7fff addrs → stack overflow; installers/winecfg/prefix-ops now run `plainGraphics=true` (no d3d overrides) via WineRunner.run(plainGraphics:). (b) wedged wineserver from a prior crash breaks all later runs with same mmap errors → runInstaller kills the bottle's wineserver first. (c) cross-version prefix (created on 7.7, switched to 11) crashes → setEngine runs `wineboot -u`. Verified: Notepad++ NSIS installer installs clean on Wine 11 plain-graphics (exit 0). Rule: NEVER apply game renderer overrides to installers/tools — only to actual game launches.
- REAL root cause of THPS repack crash (2026-07-06): setup.exe is 32-bit InnoSetup. Gcenx wine-staging 11.10 uses "new WoW64" → 32-bit RtlUnwindEx unwind loop → stack overflow (crashes at LOAD, 1 log line). GPTK 7.7 dies earlier on decompression (alloc_pages_vprot). CrossOver's patched wine handles 32-bit properly. My "Wine 11 fixes it" was WRONG — based only on the decompression stage, never a completed install.
- CrossOver engine (PR #5): EngineManager.detectCrossOver() auto-detects /Applications/CrossOver.app and exposes its `bin/wineloader` as a selectable engine (kind .crossover, D3DMetal/DXVK/WineD3D), driven against GamePorter's OWN prefixes (NOT CrossOver bottles) via Engine.extraEnv = WINELOADER/WINESERVER/WINEDLLPATH(lib/wine:lib64/wine)/DYLD_FALLBACK_LIBRARY_PATH(lib:lib64), applied in WineRunner.environment. The perl `bin/wine` wrapper needs a CX bottle ("bottle 'default' not found") — must use `bin/wineloader` binary directly. Verified: installs THPS repack, extracts C:\THPS\{Base,Engine,_Redist} 376MB+, no crash. Needs a valid CrossOver license (user's own install, personal use).
- "Access denied" install fix (PR #6): FitGirl/repack installers default the install dir to the drive with MOST FREE SPACE. Z: maps to macOS root / (huge free space) → they pick Z:\Games = / → Error 5 Access denied. Fix: mark Z: as network drive (`reg add HKLM\Software\Wine\Drives /v z: /t REG_SZ /d network /f`) → installers skip it, default to C:\Games (inside bottle, writable). Verified via InnoSetup /LOG. WineRunner.configureDrives() called in createPrefix/updatePrefix + before each installer (self-heals old bottles). SystemDrive is correctly C: — the issue is free-space heuristic, not {sd}.
- REPACK/SCENE INSTALL WORKFLOW (works, 2026-07-06): FitGirl FreeArc installer HANGS under ALL Wine-on-Mac tools (unarc.dll deadlocks under Rosetta — confirmed CrossOver-the-app hangs identically; 100% CPU, 0 bytes read). - Game launchers (PR #7, CrossOver-style): discoverGames() parses Desktop/Start-Menu .lnk (Shell Link LocalBasePath at LinkInfo+16) → maps Win path to bottle → shows Play launcher in Games section. prettyName() cleans SCENE-CAPS. + Add Game… button (pin any exe). + reload() recovers bottles that lost gameporter.json (rebuilds metadata from the drive_c prefix — a bottle DID lose its json and vanished while the 23GB game sat safe on disk; game installs to C:\Games so also added that to discoverPrograms scan).
- Gotcha: quitting GamePorter via `osascript quit` may not fully relaunch (focuses old instance) — `killall -9 GamePorter` before `open -a` when testing new builds. Bottle folder name = UUID = Bottle.id.uuidString (uppercase).
